# 狼人杀 Werewolf — 游戏规则与架构 (BRD)

本文件是项目的需求与设计基准 (BRD)：先把**游戏规则**讲清楚，再定义**软件架构**。
技术实现细节默认交由开发完成；本文档的权威内容是「规则」与「分层约定」。

> 远期：架构按「规则引擎」分层，未来或可复用到其它规则类游戏。但**当前阶段只聚焦狼人杀**，不为通用化做任何额外抽象。

---

# 第一部分：游戏规则 (Game Rules)

## 1. 名词与阵营

- **阵营 Faction**：`狼人 (Wolf)` / `好人 (Town)`。好人内部再分 `神 (PowerRole)` 与 `平民 (Civilian)`。
- **存活状态 Status**：`存活 (Alive)` / `出局 (Out)`。出局玩家不再发言、不投票、不触发技能（除死亡触发类技能，如猎人开枪）。
- **死因 DeathCause**：`狼杀 (Killed)` / `毒杀 (Poisoned)` / `放逐 (Exiled)` / `枪杀 (Shot)` / `自爆 (BlownUp)`。死因影响后续判定（例如「被毒不能开枪」）。
  - **一名玩家可同时记录多个死因**：例如同夜既被狼刀又被女巫毒，两个死因都要记录下来（见 §5.2）。

## 2. 角色定义 (Roles)

### 狼人 Werewolf（阵营：狼人）
- 夜晚集体睁眼，狼队**统一**选择刀杀一名玩家；可选择**空刀**（不杀人）。
- 刀杀目标可为**任意存活玩家，包括狼队自己人（自刀）**——自刀常用于触发狼枪、做演技等。
- 狼人之间互相知道队友身份。
- **自爆**（配置 `blownUpEnabled`，默认开）：白天发言阶段，任一狼人可主动自爆出局（死因=自爆）。自爆后**当天白天立即结束**，跳过剩余发言与放逐投票，直接进入黑夜。自爆狼不留遗言。
  - **自爆狼的当夜参与（仅讨论，无机制影响）**：自爆狼当夜仍可参与该夜狼队刀人的讨论，待狼人闭眼后再离场——这只是线下流程，**对引擎结算无影响**（夜刀仍是狼队一次决策）。
  - **最后一狼自爆**：仍**直接判负**（无狼可刀）。〔机械狼板边际：若自爆的是除机械狼外的最后一只小狼，小狼**直接离场、不参与点刀**，不能向机械狼传信息——因机械狼本就独立行动，引擎层无差异。〕

### 狼枪 WolfGun（阵营：狼人）—— 12 人板
- 是一名**狼人**：参与夜刀、与狼互通身份、白天可自爆，计入「可睁眼狼人」（绑票/拍刀口径）。
- **死亡开枪**：出局时可开枪带走一名存活玩家（死因=枪杀），机制同猎人翻枪（始终在白天结算）。
- **开枪条件**：
  - **不能**开枪：**自爆**、或**死因含毒杀**。
  - **可以**开枪：**放逐**、**被刀（含被自刀）**、**被枪杀**。
- 与猎人**复用同一件「死亡触发开枪」能力**，仅「禁枪条件」不同（猎人：带毒；狼枪：带毒 **或** 自爆）。

### 机械狼 MechanicWolf（阵营：狼人）—— 通灵机械狼板
- 是一名**狼人**：计入「所有狼人出局 → 好人胜」的胜负判定；但**不与狼队见面**，且**不计入「可睁眼狼人」**——故**不参与 §4.3 绑票计数、也不计入 §4.4 拍刀的 N**（属「不睁眼狼」）。
- **独立刀（天生，与学习无关）**：当**其他狼人全部出局**后，机械狼每晚可睁眼刀一人（普通狼刀）。
- **学习（全局仅一次，任意夜，目标须为存活玩家）**：指定一名存活玩家，学习其身份并获得对应能力。
  - **生效时机**：学习当晚**不发动**；主动技能**最早从下一晚（把学习当晚记为第 1 晚，则下一晚=第 2 晚）起**生效。**唯独「伪装」当晚即生效**。
  - **伪装（被动，当晚即生效）**：被通灵师查验时显示为**所学身份**；**未学习**则显示为「机械狼」。
  - **学习平民**：无技能（仅伪装成平民）。
  - **学习通灵师**：从学习后的第 2 晚起，每晚查验一名玩家的**具体身份**。
  - **学习女巫**：获得**学习当刻女巫剩余药水的独立副本**（女巫若已用解药则只得毒药，余类推）；与女巫**各自独立**使用、互不影响（场上可同时存在两瓶毒等）。
  - **学习猎人**：被刀 / 被放逐出局时可开枪带人，**带毒不能开枪**（复用猎人「死亡触发开枪」）。
  - **学习狼人**：在「其他狼全灭、机械狼睁眼刀人」时**额外获得一次性「大刀」**——大刀**破盾**（无视守卫守护），可留至任一晚发动、仅一次；**发动大刀的那晚共刀两人**（大刀目标 + 普通刀目标，各带走一人）。
  - **学习守卫**：获得守护能力（同样受「不可连守」与「同守同救=死（仅对狼刀）」约束）；**机械狼专属反弹**：若其守护的玩家被女巫**毒** → **女巫被反弹致死、被守玩家存活**（区别于真·守卫：真守卫挡不住毒、亦无反弹）。

### 预言家 Seer（阵营：神 / 好人）
- 每晚查验一名玩家，私下得知该玩家是 **好人 (金水)** 还是 **狼人 (查杀)**。
- 结果只有预言家本人（系统私下）知道。

### 通灵师 Psychic（阵营：神 / 好人）—— 通灵机械狼板
- 每晚查验一名玩家，私下得知其**具体身份/角色**（不是只「好/坏」，而是确切角色名，如「女巫」「狼人」「机械狼」）。
- 结果仅通灵师本人（系统私下）知道。
- 对机械狼的查验依其**伪装**显示（已学习→显示所学身份；未学习→显示「机械狼」，见 §2 机械狼）。

### 女巫 Witch（阵营：神 / 好人）
- 全程持有**解药 ×1**、**毒药 ×1**，各仅一次。
- **解药**：可救回当晚被狼刀的玩家。
  - **死讯可见性**：只有在**解药尚未使用**时，女巫每晚才会被告知「今晚谁被狼刀」；**一旦解药已用过，之后的夜晚不再获知被刀者**。
  - 但法官（系统）仍会**完整走完询问女巫的流程**（即使她已无解药、无法得知信息），以防其他玩家通过流程节奏/时长推断出信息。
  - **全程不可自救**（配置 `witchSelfRescue`，默认 `Never`）。
- **毒药**：毒杀一名存活玩家，当晚死亡（死因=毒杀）。
- **每晚仅可使用一瓶药**（配置 `witchBothPotionsSameNight`，默认 `false`：同一晚解药、毒药只能用其一）。
- 被女巫毒杀的玩家**不能触发猎人开枪**（见猎人）。
- **「同守同救 = 死」**：若守卫的守护与女巫的解药**同夜作用于同一玩家**，该玩家**仍然死亡**（见守卫 §2 与结算 §5.2）。9 人板无守卫；12 人守卫板生效。

### 猎人 Hunter（阵营：神 / 好人）
- 出局时可**开枪**带走一名存活玩家，被带走者立即出局（死因=枪杀）。
- **例外**：当死因**包含毒杀**时，猎人**不能**开枪（即使同时也被狼刀，只要带毒就不能开枪）；死因为狼杀、放逐时**可以**开枪。
- **开枪时机始终在白天**：无论猎人是**夜晚被刀**还是**白天被投/被枪出局**，开枪（翻牌）都在**白天**进行——夜晚被刀的猎人于**次日天亮公布死讯时**翻枪，**绝不在夜里开枪**。（「翻枪」是一个白天动作，夜晚死亡只是把它推迟到天亮结算。）
- **每晚验枪**：每晚法官会**私下告知猎人当前是否可以开枪**（即是否被禁枪/带毒）。这样猎人若出局，可据此决定是否动用技能。
- **出局时的选择（信息隐藏）**：猎人出局时可选择
  - **不翻牌直接离场**：不亮身份、不开枪，不暴露任何信息；或
  - **翻牌（亮明猎人身份并尝试开枪）**：若此时可开枪则带走一名玩家；若不可开枪，则被告知「**无法发动技能**」，但**法官不说明原因**（不暴露是因为被毒），以免泄露女巫用毒等信息。
- 开枪造成的死亡同样进入死亡结算（引擎须支持死亡的连锁触发，尽管本板只有一名猎人）。

### 守卫 Guardian（阵营：神 / 好人）—— 12 人板
- 每晚**守护一名玩家**，使其当晚免疫**狼刀**；可**空守**（不守）、可**自守**（守自己）。
- **不可连续两晚守护同一玩家**（含自守；配置 `guardConsecutiveSameTarget`，默认 `false`）。首夜无「上一晚」，任意守护合法。
- **同守同救 = 死**：若守卫守护 + 女巫解药**同夜作用于同一玩家**，该玩家**仍然死亡**（§5.2）。
- **挡不住毒药**：被守护者若被女巫**毒杀** → 仍然死亡（守护只挡狼刀，不挡毒）。
- **守住被刀的女巫**（信息隐藏，§11）：若守卫守护了**仍有解药**的女巫、且该女巫当晚被狼刀：
  - 女巫**存活**（守卫救下；女巫不可自救，故无「同守同救」，仅守卫单独作用）。
  - 女巫仍会被法官告知「**今晚你被刀**」（解药未用时的死讯可见性，§2），但**不会被告知是守卫救的**——她无从分辨自己是死是活，由天亮结算揭晓。

### 平民 Civilian（阵营：好人 / 平民）
- 无技能，仅靠发言与投票。

## 4. 胜负判定 (Win Condition)

判定时机：**每例死亡结算之后** 以及 **每轮放逐之后** 立即检查。

### 4.1 基本胜负条件
- **屠边 (KillSide，默认)**：
  - 好人胜：所有**狼人**出局。
  - 狼人胜：所有**神**出局 **或** 所有**平民**出局。
- **屠城 (KillAll，可配置)**：
  - 好人胜：所有狼人出局。
  - 狼人胜：所有好人（神 + 民）出局。

### 4.2 顺序结算 —— 不存在「同时满足」
- 死亡结算是**严格顺序**的：每处理完**一例死亡**就**立即检查胜负**；一旦某方达成胜利条件，**游戏当场结束**，**尚未执行的死亡触发技能（如猎人翻枪）不再触发**。
- 因此**不会出现「同时满足双方胜利条件」**的情形（故删去旧的「好人胜优先」裁定）。
- **示例**：场上仅剩 1 狼、1 好（好人是猎人）。夜里狼刀猎人 → 天亮猎人出局 → 此刻好人全灭，**狼人胜、游戏立即结束**；猎人**没有翻枪机会**（翻枪是其后的白天动作，但游戏已经结束）。

### 4.3 狼人绑票胜利 (Vote-binding Victory)
- 当**场上「可睁眼狼人」数量 ≥ 存活好人数量**时，狼人达成**绑票胜利**——狼队已掌握票数控制权，好人无法翻盘，直接判狼人胜。
- **触发方式：引擎自动检测**（与 §4.4 拍刀的「狼人主动发起」相对）。绑票胜利和基本胜负一样，在每次胜负检查时机（§4 判定时机）由引擎自动判定，无需狼人声明。
- 计数口径：**只计「可睁眼狼人」**，**不睁眼狼不计**。已知不睁眼狼：**机械狼**（其他狼存活时不睁眼，故不计入绑票）、（未来）隐狼等。

### 4.4 拍刀 (Forced-win Declaration —— 提前结算)
**触发方式：狼人主动发起**（与 §4.3 绑票胜利的「引擎自动检测」相对）。狼人可在**白天任意时刻**宣布「拍刀」，要求提前结束并直接判定胜负。判定方式 = **明牌走一遍强制流程**：

- 设场上有 **N 名可睁眼狼人**（**不含机械狼等不睁眼狼**），则 **N-1 名狼人按宣布的顺序依次自爆**，**每次自爆带走（刀）一名玩家**；狼人须**按顺序报出拍刀目标序列**。
- 因为是**明牌**（信息公开），好人方（尤其神职）相当于**开卷考试**，可用**最优解**应对（如猎人开枪、女巫用药等）。
- **结果判定**：按该序列**模拟结算到底**——若最终仍为**狼人胜利**，则**拍刀成功、狼人获胜**；否则**拍刀失败、好人获胜**。
- 本质：拍刀是一次**提前结算**——把后续必然发生的强制流程预演一遍，直接得出结果。

> 工程提示：4.4 要求引擎的「死亡结算 + 胜负判定」能在**假设/沙盒模式**下跑一遍而不污染真实对局状态（即可对 GameState 做可回滚的推演）。

## 5. 游戏总流程 (Game Flow)

```
首夜 (Night 1)
  → 第一天白天 (Day 1)：警长竞选 → 公布死讯 → 遗言 → 发言 → 放逐投票 → 结算
  → 黑夜 (Night n)
  → 白天 (Day n)：公布死讯 → 遗言 → 发言 → 放逐投票 → 结算
  → ... 直到满足胜负条件
```

### 5.1 夜晚 (Night) 行动顺序
> 顺序**与板子相关**；下为 9 人 / 12 守卫板的默认顺序，通灵机械狼板的顺序见 §3。
1. **守卫**睁眼，守护一人（或空守）。〔仅守卫板〕
2. **狼人**睁眼，统一选定刀杀目标（或空刀，可自刀）。
3. **预言家**睁眼，查验一人，私下得知好/坏。
4. **女巫**睁眼，被告知今晚被刀者，决定是否用解药；再决定是否用毒药（受 §2 配置约束）。
5. **猎人开枪手势**（私密通知，非决策）：法官于女巫行动后，私下告知猎人**当前是否可开枪**（是否带毒/禁枪），供其出局时决断（§2 每晚验枪）。
6. **天亮**，统一结算夜晚死亡（见 5.2）。

### 5.2 夜晚死亡结算
- 当晚所有死亡视为**同时发生**，天亮一并公布。
- **被狼刀者 T 的存活判定**（设 `guarded` = 守卫守护了 T，`witchSaved` = 女巫对 T 用了解药）：
  - **T 死亡 ⟺ `guarded == witchSaved`**。即：两者都没作用 → 死（狼杀）；**同守同救**（都作用）→ 死；只有守卫**或**只有解药其一作用 → 存活。
  - 无守卫的板子，`guarded` 恒为假 → 退化为「用解药则活，否则死」。
- 被女巫毒者 → 出局（死因=毒杀）。**守护挡不住毒**：被守护者若被毒仍死。
- **同刀同毒**：同一人既被狼刀又被女巫毒 → **必死**，且**两个死因都记录**（狼杀 + 毒杀）。因死因含毒杀，若该玩家是猎人/狼枪则**不能开枪**。
- 死亡触发类技能（猎人开枪）在死亡公布后按 §2 规则结算，可能引发连锁死亡，连锁需继续结算至无新死亡为止。
- **顺序与中断**：连锁结算按顺序逐例处理，**每例死亡后立即检查胜负**；一旦分出胜负则**当场结束、不再触发后续技能**（见 §4.2，如「最后一刀杀掉最后一个好人，猎人来不及翻枪」）。

### 5.3 白天 (Day) 行动顺序
- **第一天**：警长竞选（§7，仅 `sheriffEnabled` 时）→ 公布昨夜死讯 → 死者遗言 → 警长安排发言顺序 → 依次发言 → 放逐投票（§6）→ 票出者遗言与技能结算。
- **第二天起**：公布死讯 → 遗言 → （警长安排发言顺序）→ 发言 → 放逐投票 → 结算。
- **遗言规则**（统一判定，**按「出局时机 + 死因」派生，不针对具体角色/死法单独开关**）：
  - 判定式：`hasLastWords = (出局发生在白天 且 死因 ≠ 自爆) 或 (出局发生在夜晚 且 为第一夜)`
  - 等价说明：白天以**任何形式**出局（放逐、白天被猎人枪杀等）→ 有遗言；狼人自爆 → 无遗言；夜晚出局仅第一夜有遗言（第二夜起无）。
  - **实现一致性要求**：遗言资格必须由上述**统一判定函数**得出，**不得**为「被猎人枪杀者」等情形单独设置 hasLastWords 标志。例如「白天被猎人枪杀者有遗言」是因为它满足「白天 且 非自爆」，而**非**因为它是枪杀——枪杀只是恰好落入该判定。

## 6. 放逐投票 (Exile Vote)

- 所有**存活**玩家各投一票，投给一名存活玩家；`abstainAllowed=true` 时可弃票。
- **警长票权**：放逐**首轮**警徽计 **1.5 票**，其余玩家 1 票；进入**决胜轮 (Runoff) 时警徽计 1 票**（见 §7.1 归票）。
- 票数最高者被放逐出局（死因=放逐），随后结算其死亡触发技能（如猎人开枪）。
- **平票处理** (`exileTieRule = RunoffThenNoExile`)：最高票并列者进入决胜轮 (Runoff)——并列者依次发言，由**其余存活玩家**（非并列者）重新投票（此轮警徽计 1 票）；若决胜轮仍平票，则本轮**无人出局**。无单独的警长裁决（§7.1）。

## 7. 警长竞选 (Sheriff Election) —— 重点

警长（警徽持有者）是好人/狼人都可竞争的额外身份，带来投票与流程主持的优势。竞选在**第一天白天最开始**进行（`sheriffEnabled=true` 时）；若被自爆中断，可顺延至第二天（见 7.4 / 7.5）。

### 7.1 警徽 (Badge) 的作用
1. **归票（警长的主动行为）与投票权重**：所有玩家发言结束后，警长**主动「归票」**，二选一：
   - **归单人**：警长把警徽票投向某一名玩家 → 警徽计 **1.5 票**。归单人**即表示警长已投出此票**（此情形下警长**不可弃票**）。
   - **归多人 PK**：警长把多名玩家带入决胜轮 → 警徽降为 **1 票**，且此情形下警长**可以弃票**（可不投）。
   - 决胜轮中警徽一律计 **1 票**。
2. **主持发言顺序**：警长决定白天发言的起点与方向：
   - **单死**（昨夜仅 1 人出局）→ 可选择从死者的**左**或**右**邻座开始发言（俗称「死左 / 死右」）；
   - **多死或平安夜**（≥2 人出局，或无人出局）→ 从**警长自己**的左或右开始发言。
3. **无单独的平票裁决**：警长不另行裁决平票；平票完全由上述「归票 / 警徽票权」机制体现（见 §6）。
4. 警徽是**唯一**的：**任一时刻至多一名警长**。但警长出局后**可移交警徽更换警长**（见 §7.6），因此整局过程中警长可以易主——「唯一」约束的是**同一时刻**，而非整局只能有一人。

### 7.2 竞选流程（第一天白天开局，**先于公布夜晚死讯**）
1. **报名上警**：法官宣布竞选开始，每名存活玩家选择「上警（参选）」或「不上警」。
   - 上警者成为**候选人**；不上警者成为**普通投票人**。
2. **候选人发言**：所有候选人依次发言拉票。
3. **退水 (Withdraw) —— 任意时刻可退水**：候选人可在**任何时间**宣布退水退出竞选，**包括在其他候选人发言期间**。退水者**放弃当选资格**，且**不参与**警长投票。
4. **退水 / 自爆窗口**：所有候选人发言完毕后，专门留出一段**统一的退水 / 自爆时间**。
   - 警上若有狼人，可在此（或任意时刻）选择**自爆（吞毒）** → 触发 7.4 的自爆中断。
5. **警长投票**：**仅未上警的普通投票人**投票，每人 1 票，投给剩余候选人。
   - 候选人、退水者均**不投票**。
6. **结果**：
   - 若退水后**仅剩一名**候选人 → **自动当选**，无需投票。
   - 否则得票最高者当选警长；当选后由其指定第一天发言顺序（起点 + 方向）。

### 7.3 竞选边界规则
- **无人上警** → 本局无警长。
- **全员上警** → 警徽**直接流失**，本局无警长。
- **全员退水** → 本局无警长。
- **退水后仅剩一人** → 自动当选（见 7.2-6）。
- **竞选平票**：最高票并列时，并列候选人进入**一轮决胜轮 (Runoff) 发言**；决胜轮发言后，**除决胜轮候选人外的所有存活玩家**重新投票（注意：决胜轮投票范围比首轮更广——首轮仅未上警者，决胜轮为除决胜轮候选人外全体）；若**再次平票** → 警徽**流失**，本局无警长。

### 7.4 自爆中断竞选（第一天）
- 警长竞选**完成之前**，若有狼人（无论是否在警上）**自爆**：
  1. 法官**先宣布昨夜死亡信息**（竞选先于公布死讯，此时尚未公布，需补宣布）；
  2. 竞选**当场中止**，当天白天结束，**直接进入黑夜**（无放逐投票）。
- 第一天因自爆中止竞选后，警长竞选**顺延至第二天**（见 7.5）。

### 7.5 竞选顺延至第二天
- 第二天白天**直接进行警长投票**（**无发言环节**，规则同 7.2 的投票部分）。
- 若第二天**又有狼人自爆** → 竞选再次中止，**本局游戏永久不再设警徽**（此后无警长、无 1.5 票、无移交）。
- 第二天成功选出警长后，恢复正常白天流程。

### 7.6 警徽移交 (Badge Transfer on Death)
警长出局时（任何死因），在其死亡结算阶段，警长可**二选一**：
1. **移交警徽**给任一**存活**玩家（该玩家成为新警长，继承 1.5 票权与主持权）；或
2. **撕毁警徽**（警徽销毁，此后全局**无警长**，不再有 1.5 票与移交）。

移交细则：
- 移交/撕毁发生在警长**死亡公布时**（夜晚被刀 → 次日天亮公布时移交；被放逐 / 被毒 / 被枪杀 → 该死亡结算时移交）。
- **结算顺序**：当警长**同时**触发移交与其它死亡技能时，**先移交警徽，再结算技能**。例：警长是猎人被放逐 → 先移交警徽，再开枪。
- 新任警长立即拥有完整警长权能（含后续若其出局时的再移交权）。
- 撕毁后不可恢复。

# 第二部分：软件架构 (Architecture)

> 仅作分层约定；具体实现细节交由开发。原则：**规则数据化、流程显式化、I/O 与逻辑解耦**。

## 8. 领域模型

### 8.0 角色建模的核心决策：浅继承 + 能力组合（composition over deep inheritance）

角色沿**多个独立维度**变化（阵营 × 子类型 × 能力组合），单链深继承装不下多轴，遇到「白狼王 / 隐狼 / 机械狼」这类 mix-and-match 会断裂（典型反例：隐狼*是*狼却**不参与夜刀**，深继承只能靠重写去「删」父类能力）。因此约定：

- **不做** `Role → Town → PowerRole → Seer` 这样的深层嵌套继承。
- **阵营 / 子类型用数据（enum）**，不做行为类层级——`Town` 这层没有共同行为（平民无技能、投票发言人人都有），建空类是「为继承而继承」。
- **多态主要落在「能力 Ability」这一粒度**：角色 = 元数据 + **一组可复用能力组件**。加新角色 ≈ 挑选/新写能力再拼装，基本是「加数据」。
- **通用玩家行为（投票 / 发言 / 竞选警长）属于 `Player` 与流程层，不属于 `Role`**。

### 8.1 各模型职责

- **Player**：玩家本体与「人人都有」的能力。
  - 字段：`id / name / seat`、`status`（`Alive/Out` + 死因集合 + 死亡天数）、`role`（`std::unique_ptr<Role>`）、瞬态标记（本夜被守护/中毒、是否警长等）。
  - 通用行为：投票、发言、竞选/退水/归票——**与角色无关**，由流程驱动。

- **Faction（枚举，数据）**：`Wolf / Town`，`Town` 内再分子类型 `PowerRole / Civilian`。胜负判定（§4）是**全局按阵营计数**得出，不是角色身上的方法。「可睁眼狼人」等口径也由此数据派生。

- **Role（轻量，组合能力）**：一个具体角色一个类型（`Werewolf / WolfGun / MechanicWolf / Seer / Psychic / Witch / Hunter / Guardian / Civilian`，未来 `WhiteWolfKing / HiddenWolf / …`）。
  - 持有：`name`、`faction`、`subKind`、`abilities: vector<unique_ptr<Ability>>`。
  - Role 本身**扁平**（不深嵌套）；行为不靠继承父类，而靠**装配能力**。

- **机械狼（架构压力测试，待 M9）**：是组合式能力体系最硬的检验——
  - **运行时学习身份** = 在游戏中**动态加挂**所学角色的能力（学女巫→加一个带**独立药水库存**的 `WitchPotions`；学守卫→加 `Protect` 的反弹变体；学狼人→给独立刀加「大刀」修饰）。
  - **伪装** = 一个**被动信息覆盖**钩子（覆盖通灵师查验返回的身份），与「学到的主动技能下一晚才生效」分离。
  - **不睁眼狼** = 阵营计数需区分「是否计入可睁眼狼」（§4.3/§4.4）。
  - 这些都顺着既有抽象走（能力组件 + 信息钩子 + 阵营数据），不需要改引擎主干——正是「加角色≈加数据/能力」的极限检验。

- **Ability（多态基类，多态的主战场）**：一个能力一个子类，挂到引擎的行为钩子上。例：
  - `NightKill`（狼队共享）、`Inspect`（预言家）、`WitchPotions`（女巫）、`Protect`（守卫夜间守护）、
  - `DeathTriggerShoot`（猎人 / 狼枪 / 白狼王 **复用同一件**，以「禁枪死因集合」参数化：猎人={毒}，狼枪={毒,自爆}）、`SelfExplode`（所有狼共享）、`HiddenFromSeer`（隐狼）……
  - 钩子类型大致为：**夜间行动**、**死亡触发**、**被动/修正**、**信息下发**（按 §11 定向）。
  - 装配示例：狼枪 `= {Wolf, [NightKill, SelfExplode, DeathTriggerShoot(禁:{毒,自爆})]}`；守卫 `= {Town/PowerRole, [Protect]}`；猎人 `= {Town/PowerRole, [DeathTriggerShoot(禁:{毒}), NightGunCheck]}`。
  - **守卫状态**：`Protect` 需记录「上一晚守护对象」以实现「不可连守」（放 `GameState` 或能力内）；夜晚结算按 §5.2 的 `guarded == witchSaved` 公式判定狼刀目标存亡。

- **GameState**：全局状态——玩家列表、当前阶段、天数、警长是谁（可变单值引用，§7.1）、女巫药剂剩余、夜晚待结算缓存、事件/历史日志。需支持**可回滚推演**（沙盒）以服务拍刀提前结算（§4.4）。

- **Board**：板子配置（角色构成 + §3 配置项 + 胜负规则）。新增板子主要是「加数据」。

## 9. 流程与判定

- **PhaseFlow**：把 §5 的流程建模为显式状态机/步骤序列（Night → Day(竞选/公布/发言/投票) → …）。每步：收集决策 → 应用 → 结算事件 → 检查胜负。
- **WinCondition**：实现 §4，每次死亡/投票后调用，返回「未结束 / 好人胜 / 狼人胜」。
- **死亡结算**：统一入口，处理「被刀/被毒/被放逐/被枪杀」，并驱动死亡触发技能的**连锁结算**直至稳定。

## 10. I/O 解耦

- 引擎为**纯逻辑**，需要玩家决策时通过 `DecisionProvider` 接口请求（「谁要选、合法选项是什么」），绝不直接读写控制台。
- 实现：`ConsoleDecisionProvider`（真人终端）、`ScriptedDecisionProvider`（测试用预设脚本）、（后续）`BotDecisionProvider`（自动模拟）。
- 这样同一套规则逻辑可被「单测、命令行对局、批量模拟」共用。

## 11. 信息可见性 (Information Visibility) —— 核心原则

引擎必须严格区分两层，**绝不能把「真相」直接暴露给玩家**：

- **真相层 (GameState)**：客观发生的一切（谁被刀、谁带毒、谁是狼……）。
- **玩家视图层 (Player View)**：每个玩家**被允许知道**的信息子集，按玩家定向下发。

由此衍生的规则（均已在第一部分出现）：
- 预言家查验结果仅其本人可见。
- 女巫用掉解药后，不再获知夜晚死讯（§2）。
- 猎人每晚私下验枪；出局选择翻牌却不能开枪时，仅告知「无法发动技能」、**不报原因**（§2）。
- 狼队夜晚刀人目标仅狼队可见。

**关键工程约束——「流程节点」与「信息暴露」解耦**：即使某玩家这一步拿不到任何信息（如女巫已无解药），法官**仍要照常走完该节点**（相同提示、相近耗时），以免其他玩家从「跳过 / 节奏 / 时长」反推出隐藏信息。`DecisionProvider` 的 `notify` 必须**按玩家定向**，且「是否触发节点」独立于「是否真的下发信息」。

### 产品形态路线图（远期，先记录后做）

当前程序是「**法官单机工具**」：法官一人操作，cue 完整流程、录入玩家行为、引擎判定。后续逐步演进为多人/AI 对战平台：

1. ✅ **法官模式**：开局录入身份、状态面板、遗言/发言顺序 cue、主持叙述与节奏停顿。让法官拿着一台机器即可主持整局。
2. ✅ **随机发牌**：程序随机分配身份（替代手动录入），法官只看分配结果。
3. **多人联机**：法官 + 各玩家各自通过程序（多端/网络）参与，**每个玩家只看到自己被允许的信息**（真正落地 §11 的「按玩家定向下发」，不再是单屏 moderator 模式）。
4. **发言记录**：支持玩家发言（文本/语音转写），程序为每位玩家**留存发言记录**，可复盘。
5. **AI Agent 玩家**：可接入不同 AI agent 作为玩家加入对局，与真人/彼此博弈（复用 `DecisionProvider` 接口：`BotDecisionProvider` / `AgentDecisionProvider`）。

> 架构上这些都顺着既有抽象走：玩家行为统一经 `DecisionProvider`；信息下发统一经 `notify`（按玩家定向）；引擎纯逻辑不变。形态演进主要是「换 I/O 实现 + 加网络层」，规则引擎应保持稳定。
